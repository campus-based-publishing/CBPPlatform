00019  *  OpenURL, dc(terms), MODS) or a subset of it.
00020  *
00021  *  We only implement such subsets of meta-data schemas that contain elements which
00022  *  can be mapped to PKP application objects. Meta-data schemas are not meant to
00023  *  represent any meta-data in the given schema just PKP application meta-data. The
00024  *  constructor argument uniquely identifies the application objects this meta-data
00025  *  schema can be mapped to. There should never be two MetadataSchemas with the same
00026  *  namespace that map to the same application object type. This also means that we
00027  *  implement composite elements if and only if the composite complies with our
00028  *  internal class composition schema and not only because the schema allows a composite
00029  *  in a certain position. See MetadataDescription and MetadataProperty for further
00030  *  information about composite meta-data properties.
00031  *
00032  *  Example: We implement a composite to represent authors that correspond to the
00033  *  PKPAuthor class. We do not implement composites for title meta-data
00034  *  even if the chosen schema allows this (e.g. abbreviated title, alternative title)
00035  *  as this data is implemented as fields of the Submission object. This doesn't mean
00036  *  that such data cannot be mapped to composites in external bindings, e.g. in an
00037  *  XML binding of the meta-data schema. We can always map a flat list of key/value
00038  *  pairs to a hierarchical representation in an external binding.
00039  *
00040  *  This coupling allows us to flexibly configure meta-data entry for application
00041  *  objects. We can identify appropriate meta-data fields for application objects
00042  *  when displaying or entering object meta-data in application forms. Thereby users
00043  *  can dynamically add meta-data fields for input/output if they require these for
00044  *  their local meta-data partners (e.g. libraries, repositories, harvesters, indexers).
00045  *
00046  *  We assume that all properties defined within a schema can potentially be assigned
00047  *  to the objects represented by the given association types. Users should, however,
00048  *  be able to enable / disable properties on a per-assoc-type basis so that only a
00049  *  sub-set of properties will actually be available in the user interface as well as
00050  *  exported or imported for these objects.
00051  *
00052  *  New schemas can be dynamically added to the mix at any time if they provide fields
00053  *  not provided by already existing schemas.
00054  *
00055  *  NB: We currently provide meta-data schemas as classes for better performance
00056  *  and code readability. It might, however, be necessary to maintain meta-data
00057  *  schemas in the database for higher flexibility and easier run-time configuration/
00058  *  installation of new schemas.
