00015  * @brief Class modeling a description (DCMI abstract model) or subject-
00016  *  predicate-object graph (RDF). This class and its children provide
00017  *  meta-data (DCMI abstract model: statements of property-value pairs,
00018  *  RDF: assertions of predicate-object pairs) about a given PKP application
00019  *  entity instance (DCMI abstract model: described resource, RDF: subject).
00020  *
00021  *  This class has primarily been designed to describe journals, journal
00022  *  issues, articles, conferences, conference proceedings (conference papers),
00023  *  monographs (books), monograph components (book chapters) or citations.
00024  *
00025  *  It is, however, flexible enough to be extended to describe any
00026  *  application entity in the future. Descriptions can be retrieved from
00027  *  any application object that implements the MetadataProvider interface.
00028  *
00029  *  Special attention has been paid to the compatibility of the class
00030  *  implementation with the implementation of several meta-data standards
00031  *  that we consider especially relevant to our use cases.
00032  *
00033  *  We distinguish two main use cases for meta-data: discovery and delivery
00034  *  of described resources. We have chosen the element-citation tag from the
00035  *  NLM standard <http://dtd.nlm.nih.gov/publishing/tag-library/3.0/n-8xa0.html>
00036  *  as our primary representation of delivery meta-data and dcterms
00037  *  <http://dublincore.org/documents/dcmi-terms/> as our primary
00038  *  representation of discovery meta-data.
00039  *
00040  *  Our specific use of meta-data has important implications and determines
00041  *  our design goals:
00042  *  * Neither NLM-citation nor dcterms have been designed with an object
00043  *    oriented encoding in mind. NLM-citation is usually XML encoded
00044  *    while typical dcterms encodings are HTML meta-tags, RDF or XML.
00045  *  * We believe that trying to implement a super-set of meta-data
00046  *    standards ("least common denominator" or super-schema approach)
00047  *    is fundamentally flawed as meta-data standards are always
00048  *    developed with specific use-cases in mind that require potentially
00049  *    incompatible data properties or encodings.
00050  *  * Although we think that NLM-citation and dcterms are sensible default
00051  *    meta-data schemes our design should remain flexible enough for
00052  *    users to implement and use other schemes as an internal meta-data
00053  *    standard.
00054  *  * We have to make sure that we can easily extract/inject meta-data
00055  *    from/to PKP application objects.
00056  *  * We have to avoid code duplication to keep maintenance cost under
00057  *    control.
00058  *  * We have to minimize the "impedance mismatch" between our own
00059  *    object oriented encoding and fully standard compliant external
00060  *    encodings (i.e. XML, RDF, HTML meta-tags, ...) to allow for easy
00061  *    conversion between encodings.
00062  *  * We have to make sure that we can switch between internal and
00063  *    external encodings without any data loss.
00064  *  * We have to make sure that crosswalks to and from other important
00065  *    meta-data standards (e.g. OpenURL variants, MODS, MARC) can be
00066  *    performed in a well-defined and easy way while minimizing data
00067  *    loss.
00068  *  * We have to make sure that we can support qualified fields (e.g.
00069  *    qualified DC).
00070  *  * We have to make sure that we can support RDF triples.
00071  *
00072  *  We took the following design decisions to achieve these goals:
00073  *  * We only implement properties that are justified by strong real-world
00074  *    use-cases. We recognize that the limiting factor is not the data that
00075  *    we could represent but the data we actually have. This is not determined
00076  *    by the chosen standard but by the PKP application objects we want to
00077  *    represent. Additional meta-data properties/predicates can be added as
00078  *    required.
00079  *  * We do adapt data structures as long as we can make sure that a
00080  *    fully standard compliant encoding can always be re-constructed. This
00081  *    is especially true for NLM-citation which is designed with
00082  *    XML in mind and therefore uses hierarchical constructs that are
00083  *    difficult to represent in an OO class model.
00084  *    This means that our meta-data framework only supports (nested) key/
00085  *    value-based schemas which can however be converted to hierarchical
00086  *    representations.
00087  *  * We borrow class and property names from the DCMI abstract model as
00088  *    the terms used there provide better readability for developers less
00089  *    acquainted with formal model theory. We'll, however, make sure that
00090  *    data can easily be RDF encoded within our data model.
00091  *  * Data validation must ensure that meta-data always complies with a
00092  *    specific meta-data standard. As we are speaking about an object
00093  *    oriented encoding that is not defined in the original standard, we
00094  *    define compliance as "roundtripability". This means we must be able
00095  *    to convert our object oriented data encoding to a fully standard
00096  *    compliant encoding and back without any data loss.
00097  */

00236     * Add a meta-data statement. Statements can only be added
00237     * for properties that are part of the meta-data schema. This
00238     * method will also check the validity of the value for the
00239     * given property before adding the statement.

00241     * @param $value mixed The value to be assigned to the property

00247       // Check the property
00248       $property =& $this->getProperty($propertyName);
00249       if (is_null($property)) return false;
00250       assert(is_a($property, 'MetadataProperty'));
00251 
00252       // Check that the property is allowed for the described resource
00253       if (!in_array($this->_assocType, $property->getAssocTypes())) return false;
00254 
00255       // Handle translation
00256       $translated = $property->getTranslated();
00257       if (isset($locale) && !$translated) return false;
00258       if (!isset($locale) && $translated) {
00259          // Retrieve the current locale
00260          $locale = Locale::getLocale();
00261       }
00262 
00263       // Check that the value is compliant with the property specification
00264       if ($property->isValid($value, $locale) === false) return false;
00265 
00266       // Handle cardinality
00267       $existingValue =& $this->getStatement($propertyName, $locale);
00268       switch ($property->getCardinality()) {
00269          case METADATA_PROPERTY_CARDINALITY_ONE:
00270             if (isset($existingValue) && !$replace) return false;
00271             $newValue =& $value;
00272             break;
00273 
00274          case METADATA_PROPERTY_CARDINALITY_MANY:
00275             if (isset($existingValue) && !$replace) {
00276                assert(is_array($existingValue));
00277                $newValue = $existingValue;
00278                array_push($newValue, $value);
00279             } else {
00280                $newValue = array(&$value);
00281             }
00282             break;
00283 
00284          default:
00285             assert(false);
00286       }
00287

00364    /**
00365     * Add several statements at once. If one of the statements
00366     * is invalid then the meta-data description will remain in its
00367     * initial state.
00368     * * Properties with a cardinality of 'many' must be passed in as
00369     *   sub-arrays.
00370     * * Translated properties with a cardinality of 'one' must be
00371     *   passed in as sub-arrays with the locale as a key.
00372     * * Translated properties with a cardinality of 'many' must be
00373     *   passed in as sub-sub-arrays with the locale as the first key.
00374     * @param $statements array statements
00375     * @param $replace integer one of the allowed replace levels.
00376     * @return boolean true if all statements could be added, false otherwise
00377     */
00378    function setStatements(&$statements, $replace = METADATA_DESCRIPTION_REPLACE_PROPERTIES) {
00379       assert(in_array($replace, $this->_allowedReplaceLevels()));
00380 
00381       // Make a backup copy of all existing statements.
00382       $statementsBackup = $this->getAllData();
00383 
00384       if ($replace == METADATA_DESCRIPTION_REPLACE_ALL) {
00385          // Delete existing statements
00386          $emptyArray = array();
00387          $this->setAllData($emptyArray);
00388       }
00389 
00390       // Add statements one by one to detect invalid values.
00391       foreach($statements as $propertyName => $content) {
00392          assert(!empty($content));
00393 
00394          // Transform scalars or translated fields to arrays so that
00395          // we can handle properties with different cardinalities in
00396          // the same way.
00397          if (is_scalar($content) || is_string(key($content))) {
00398             $values = array(&$content);
00399          } else {
00400             $values =& $content;
00401          }
00402 
00403          if ($replace == METADATA_DESCRIPTION_REPLACE_PROPERTIES) {
00404             $replaceProperty = true;
00405          } else {
00406             $replaceProperty = false;
00407          }
00408 
00409          foreach($values as $value) {
00410             // Is this a translated property?
00411             if (is_array($value)) {
00412                foreach($value as $locale => $translation) {
00413                   // Handle cardinality many and one in the same way
00414                   if (is_scalar($translation)) {
00415                      $translationValues = array(&$translation);
00416                   } else {
00417                      $translationValues =& $translation;
00418                   }
00419                   foreach($translationValues as $translationValue) {
00420                      // Add a statement (replace existing statement if any)
00421                      if (!($this->addStatement($propertyName, $translationValue, $locale, $replaceProperty))) {
00422                         $this->setAllData($statementsBackup);
00423                         return false;
00424                      }
00425                      // Reset the $replaceProperty flag to avoid that subsequent
00426                      // value entries will overwrite previous value entries.
00427                      $replaceProperty = false;
00428 
00429                      unset($translationValue);
00430                   }
00431                   unset($translationValues);
00432                }
00433                unset($translation);
00434             } else {
00435                // Add a statement (replace existing statement if any)
00436                if (!($this->addStatement($propertyName, $value, null, $replaceProperty))) {
00437                   $this->setAllData($statementsBackup);
00438                   return false;
00439                }
00440 
00441                // Reset the $replaceProperty flag to avoid that subsequent
00442                // value entries will overwrite previous value entries.
00443                $replaceProperty = false;
00444             }
00445             unset($value);
00446          }
00447          unset($values);
00448       }
00449       return true;
