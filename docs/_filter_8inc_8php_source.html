00011  * @brief Class that provides the basic template for a filter. Filters are
00012  *  generic data processors that take in a well-specified data type
00013  *  and return another well-specified data type.
00014  *
00015  *  Filters enable us to re-use data transformations between applications.
00016  *  Generic filter implementations can sequence, (de-)multiplex or iterate
00017  *  over other filters. Thereby filters can be nested and combined in many
00018  *  different ways to form complex and easy-to-customize data processing
00019  *  networks or pipelines.
00020  *
00021  *  NB: This also means that filters only make sense if they accept and
00022  *  return standardized formats that are understood by other filters. Otherwise
00023  *  the extra implementation effort for a filter won't result in improved code
00024  *  re-use.
00025  *
00026  *  Objects from different applications (e.g. Papers and Articles) can first be
00027  *  transformed by an application specific filter into a common format and then
00028  *  be processed by application agnostic import/export filters or vice versa.
00029  *  Filters can be used to pre-process data before it is indexed for search.
00030  *  They also provide a framework to customize the processing applied in citation
00031  *  parsing and lookup (i.e. which parsers and lookup sources should be applied).
00032  *
00033  *  Filters can be used stand-alone outside PKP applications.
00034  *
00035  *  The following is a complete list of all use-cases that have been identified
00036  *  for filters:
00037  *  1) Decode/Encode
00038  *  * import/export: transform application objects (e.g. an Article object)
00039  *    into structured (rich) data formats (e.g. XML, OpenURL KEV, CSV) or
00040  *    vice versa.
00041  *  * parse: transform unstructured clob/blob data (e.g. a Word Document)
00042  *    into application objects (e.g. an Article plus Citation objects) or
00043  *    into structured data formats (e.g. XML).
00044  *  * render: transform application objects or structured clob/blob data into
00045  *    an unstructured document (e.g. PDF, HTML, Word Document).
00046  *
00047  *  2) Normalize
00048  *  * lookup: compare the data of a given entity (e.g. a bibliographic
00049  *    reference) with data from other sources (e.g. CrossRef) and use this
00050  *    to normalize data or improve data quality.
00051  *  * harvest: cleanse and normalize incoming meta-data
00052  *
00053  *  3) Map
00054  *  * cross-walk: transform one meta-data format into another. Meta-data
00055  *    can be represented as structured clob/blob data (e.g. XML) or as
00056  *    application objects (i.e. a MetadataRecord instance).
00057  *  * meta-data extraction: retrieve meta-data from OO entities
00058  *    (e.g. an Article) into a standardized meta-data record (e.g. NLM
00059  *    element-citation).
00060  *  * meta-data injection: inject data from a standardized meta-data
00061  *    record into application objects.
00062  *
00063  *  4) Convert documents
00064  *  * binary converters: wrap binary document converters (e.g. antidoc) in
00065  *    a well-defined and re-usable way.
00066  *
00067  *  5) Search
00068  *  * indexing: pre-process data (extract, tokenize, remove stopwords,
00069  *    stem) for indexing.
00070  *  * finding: pre-process queries (parse, tokenize, remove stopwords,
00071  *    stem) to access the index
00072  */
